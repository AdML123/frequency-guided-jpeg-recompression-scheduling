# Frequency-Guided JPEG Recompression Scheduling

This repository is the public reproducibility package for the
2026 11th International Conference on Signal and Image Processing paper
`Frequency-Guided JPEG Recompression Scheduling for Adversarial Defense`.

It contains two layers:

- `results/`: the figures, table snippets, and data snapshots used in the paper.
- `src/`, `scripts/`, and `third_party/`: the code needed to regenerate derived
  figures and tables, run smoke checks, and rerun the full experiments when the
  required datasets and checkpoints are available locally.

The package is not a manuscript build tree. It does not contain the conference
LaTeX source or the submission PDF. It does contain the LaTeX table snippets
that correspond to the paper's data tables.

## Paper Claim

The paper studies the order of JPEG quality factors in multi-generation
recompression. On the included CIFAR-10 nonadaptive audit, the perturbation
discrete cosine transform centroid rule predicts whether front-loaded or fixed
recompression gives the lower attack success rate in 5/8 model-attack
conditions. The other 3/8 conditions differ by less than one percentage point.
For standard ResNet-18 attacks, changing only the schedule order changes attack
success rate by 12-19 percentage points.

## Repository Layout

- `data/derived/`: CSV and JSON files used to compute paper figures and tables.
- `results/data/`: the same final data snapshots archived with the submitted
  results.
- `results/figures/`: final PDF/PNG figures used by the paper.
- `results/tables/`: final LaTeX table snippets used by the paper.
- `src/jpeg_defense/`: schedule, JPEG, metric, plotting, model, and attack code.
- `scripts/`: entrypoints for figure/table generation, smoke checks, resource
  checks, runtime benchmarking, and full experiment reruns.
- `resources_manifest.yml`: external datasets and weights needed for full
  reruns, with source, expected path, redistribution status, and checksum notes.
- `third_party/diffjpeg/`: vendored differentiable JPEG implementation used for
  JPEG-aware boundary checks.
- `tests/`: automated checks for the package and data-result reproduction.

## Environment

The reference conda environment name in this public package is `fgjpeg`. A
different local environment name is fine as long as the same dependencies are
installed.

```powershell
conda env create -f environment.yml
conda run -n fgjpeg python -m pip install -e .
```

For a lighter CI-style install:

```powershell
conda create -n fgjpeg python=3.10 -y
conda run -n fgjpeg python -m pip install -r requirements-ci.txt
conda run -n fgjpeg python -m pip install -e .
```

## Full Experiment Reruns

Full reruns are supported by the code but require resources that are not
redistributed here:

- CIFAR-10 images from the official CIFAR-10 distribution.
- ImageNet validation images from the official ImageNet access process.
- RobustBench checkpoints and any torchvision or timm checkpoints used for
  boundary probes.
- A CUDA-capable PyTorch installation is recommended for the full attack runs.

## External Resource Policy

The external resources are listed in `resources_manifest.yml`. CIFAR-10 is
referenced by its official archive and checksum. ImageNet validation images are not redistributed because they require authorized access. RobustBench,
torchvision, and timm model weights are referenced by source and cache path;
model weights are referenced by source and cache path instead of being committed
to the git tree.

Prepare the ignored local layout:

```powershell
conda run -n fgjpeg python scripts/prepare_resources.py --data-root <local-data-root>
```

If a resource already exists locally, link or copy it into the expected layout:

```powershell
conda run -n fgjpeg python scripts/prepare_resources.py --data-root <local-data-root> --resource cifar10=<path-to-cifar-10-python.tar.gz>
```

Check the local inventory before a full rerun:

```powershell
conda run -n fgjpeg python scripts/check_resources.py
conda run -n fgjpeg python scripts/verify_resources.py --data-root <local-data-root> --hash
bash scripts/run_full_pipeline.sh
```

Machine-specific variables such as `DATA_DIR`, `RESULTS_DIR`,
`ROBUSTBENCH_ROOT`, `AUTO_ATTACK_ROOT`, and `DIFFJPEG_ROOT` should remain
outside version control.

## Data Availability

Redistributable derived data are included under `data/derived` and mirrored in
`results/data` for direct comparison with the paper results. Raw datasets and
model weights are not redistributed.

## Security

No credentials are required to inspect the included results, regenerate figures
or tables, or run smoke tests. Do not commit account secrets, private data,
model-store tokens, or local credentials.

No password authentication is used for GitHub operations. Use GitHub CLI with
`gh auth login --web`, a personal access token stored outside the repository, or
SSH.
